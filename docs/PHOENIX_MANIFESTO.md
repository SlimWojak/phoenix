## THE DOCTRINE OF FATES

*Before there were sprints, there was NEX. Before there was Phoenix, there was death.*

### üìú THE BALANCE PRINCIPLE

> "Discipline WITH innovation, not discipline AGAINST innovation.
> Governance protects the system; it should not starve the user."

This is the line Phoenix surfs:

| Discipline | Innovation |
|------------|------------|
| Quality > Speed | But "right" includes "useful" |
| Explicit > Implicit | Features must earn their place |
| Projection > Interpretation | "NEX had it" is not justification |
| Facts > Stories | Every feature must prove value |
| Human frames > System proposes | And pass the LLM removal test |
| Receipts > Narratives | And not transfer authority |

### ‚öñÔ∏è THE FOUR FATES

Every capability must answer: *"What is my fate?"*

| Fate | Meaning | Action |
|------|---------|--------|
| **KEEP** | Already aligned | Build into Phoenix |
| **REIMAGINE** | Valuable but poisoned | Redesign with discipline |
| **KILL** | Cannot be rehabilitated | Document, never resurrect |
| **PARK** | Not priority | Log for future, no sprint |

### üíÄ THE DEATH CRITERIA

A capability is KILLED when any of these are true:

```
- System authors meaning
- Embeds hidden priors
- Transfers authority from human to system
- Creates recommendation pressure
- Non-replayable without LLM
```

**The litmus test:** *If removing the LLM prevents reconstruction from raw output, the feature is invalid.*
